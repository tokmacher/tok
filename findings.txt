## Dependency Update Strategy

### Current Setup
- Dependabot: configured in `.github/dependabot.yml`, runs weekly
- Dependencies: version upper bounds in `pyproject.toml` (good)
- uv.lock: pins exact versions for deterministic builds (good)

### Recommendation: Time-Gate Package Updates

Option 1: Disable Dependabot entirely
  - Pros: Zero surprise breakage, full manual control
  - Cons: Miss security patches, stale dependencies

Option 2: Reduce Dependabot frequency to monthly or quarterly
  - Change `schedule.interval` in dependabot.yml from `weekly` to `monthly`
  - Pros: Still get updates, less churn
  - Cons: Still some risk of breaking changes

Option 3: Time-gate uv/pip resolution (your 90-day idea)
  - Add a policy that dependencies must be at least 90 days old before adoption
  - Implementation: pin versions in pyproject.toml and only bump after the gating period
  - Could automate with a script that checks release dates before allowing upgrades
  - Pros: Protects against freshly-introduced supply chain attacks and regressions
  - Cons: More manual work to maintain

Option 4: Combine approaches (recommended)
  - Set Dependabot to monthly
  - Manually review and test each bump
  - Keep upper bounds on all dependencies
  - For critical deps (httpx, openai, etc.), require 90-day lag before adopting new versions

## LOW Priority - Niche Maintainability Items

### No Changelog Automation
- CHANGELOG.md is updated manually
- Consider using `towncrier` or `keep-a-changelog` tooling for auto-generation from PR labels

### No Docs Auto-Generation
- All docs are manual markdown in docs/
- Consider adding Sphinx or MkDocs for API reference generation from docstrings
- Lower priority since the docs are well-written and current

### No tox / Nox for Local Multi-Version Testing
- CI tests on Python 3.10-3.12, but no local multi-version tool
- `tox.ini` or `noxfile.py` would let contributors test across versions locally

### CLI Without a Framework
- src/tok/__main__.py uses argparse directly
- Not a bug, but a maintainability consideration as CLI grows
- `click` or `typer` would give better help text, completion, and subcommand handling

### No Pre-Commit Hook for Tests
- .pre-commit-config.yaml runs linting and formatting
- But does not run the test suite before commits
- Consider adding a `pytest --quiet` hook for changed files

### scripts/ Directory
- Contains ad-hoc scripts (parse_tok_logs.py, tok_compress_hook.py, pr-comment.py)
- These reference hardcoded paths and may be developer-specific
- Decide whether to include, document, or exclude from the release
- If included, make paths configurable via CLI args or env vars

### data/validation_dataset.json
- 16,610 lines of synthetic validation data
- Consider whether this belongs in the repo or should be downloaded separately
- Bloats the repo size; could be hosted as a release artifact instead
