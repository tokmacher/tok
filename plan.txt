# Tok Public Release Plan

## Executive Summary
This plan outlines the remaining tasks needed to prepare the Tok repository for public open source release. The repository is substantially ready with most infrastructure in place, but several critical blockers remain.

## Critical Blockers (Must Fix)

### 1. Repository Hygiene Issues
**Status**: NEEDS WORK
- Files present that should be gitignored: `.DS_Store`, `telemetry.db`, `execution.log`, `memory.tok`, `gate-config.json`
- Internal files in repo: `CLAUDE.md`, `roadmap.md`, `human_checklist.txt`, `refactoring_plan.txt`, `release-plan.txt`, `release_ready.txt`
- `archive/` directory contains research artifacts not meant for public release

**Actions**:
```bash
# Remove files from git tracking
git rm --cached $(git ls-files | grep -E "\.DS_Store|telemetry\.db|execution\.log|memory\.tok|gate-config\.json|CLAUDE\.md|roadmap\.md|human_checklist\.txt|refactoring_plan\.txt|release-plan\.txt|release_ready\.txt")
git rm -r --cached archive/

# Ensure .gitignore is working
git status
```

**Deadline**: Before any other release activities

### 2. Test Coverage Below Target
**Status**: CRITICAL
- Current coverage: 71.39% (line-rate="0.7139")
- Target requirement: 80% (from release-checklist.md and CI)
- Gap: ~8.6% additional coverage needed

**Actions**:
- Run `pytest tests/unit tests/integration -v --cov=src/tok --cov-report=term-missing` to identify uncovered modules
- Prioritize testing modules with lowest coverage
- Focus on release surface: `src/tok/cli`, `src/tok/gateway`, `src/tok/runtime/core`

**Deadline**: Must complete before CI gate check

**Success Criteria**: Coverage ≥ 80%

    """Raised when session state is invalid."""

## Verification Checklist

Before release, ensure:

### Pre-Release Must-Haves
- [ ] All CI checks pass on `main` branch
- [ ] Test coverage reaches 80% (currently 71.39%)
- [ ] No hardcoded secrets or credentials
- [ ] Internal files removed from git tracking
- [ ] LICENSE, CODE_OF_CONDUCT, CONTRIBUTING, SECURITY files present
- [ ] Clean-room install verification works
- [ ] Bridge-first workflow validated against real Claude Code session

### Release Day Actions
- [ ] Update CHANGELOG.md with release date
- [ ] Update version in `__init__.py` and `pyproject.toml`
- [ ] Run full validation flow:
  ```bash
  pre-commit run --all-files
  python scripts/check_repo_hygiene.py
  uv run pytest tests/unit tests/integration -v -x --cov=src/tok --cov-report=term-missing --cov-fail-under=80
  uv run ruff check src/tok tests
  uv run mypy src/tok/
  python -m build
  ```
- [ ] Tag release: `git tag v0.1.0`
- [ ] Push tag: `git push origin v0.1.0`
- [ ] Monitor Release GitHub Action
- [ ] Verify PyPI publication
- [ ] Verify GitHub Release

### Post-Release
- [ ] Verify package on PyPI
- [ ] Verify README renders correctly on PyPI
- [ ] Update any internal references to new version
- [ ] Announce if appropriate

## Success Criteria

**Release is ready when:**
- All "Pre-Release Must-Haves" are checked ✅
- Test coverage ≥ 80%
- No internal files in public repository
- Clean-room install passes all README instructions
- All CI workflows are green on `main`

## Risk Mitigation

- **Coverage risk**: If coverage cannot reach 80%, document why and consider adjusting threshold or marking specific modules as excluded
- **Documentation risk**: All docs changes are reversible if issues found
- **Git hygiene risk**: All file removals can be undone with `git checkout` if needed
- **Release rollbacks**: Can delete GitHub Release and PyPI version if critical issues found

## Dependencies

- None - all work can be done by maintainer with appropriate permissions
- GitHub repository must be configured for:
  - PyPI trusted publishing
  - GitHub Pages for docs (if using)
  - Security scanning if enabled

## Notes

This plan assumes the repository is already targeting `tokmacher` organization and the codebase has been validated through internal testing cycles. The focus is specifically on making the repository suitable for public consumption.
